Phase 8 extends dmaplane from host DRAM into GPU VRAM, pinning GPU memory through the NVIDIA peer-to-peer API, mapping it through PCIe BAR1 with write-combining, registering it as an RDMA Memory Region, and sending it over real Ethernet to a second machine with no GPU.

**The `symbol_get`/`symbol_put` pattern resolves NVIDIA P2P functions at runtime without creating a static module dependency.** NVIDIA's kernel driver exports `nvidia_p2p_get_pages`, `nvidia_p2p_put_pages`, and `nvidia_p2p_free_page_table` as `EXPORT_SYMBOL_GPL`, yet nvidia.ko itself is proprietary. On kernel 6.5+, any module that imports GPL-only symbols cannot also depend on proprietary modules. Since dmaplane already depends on GPL-only symbols from `ib_core` for RDMA verbs, a static dependency on nvidia.ko causes `insmod` to fail. `symbol_get()` performs a runtime lookup in the kernel symbol table and increments the owning module's refcount without creating a modpost dependency. `dmaplane_gpu_init()` resolves all three function pointers at module load; if any fails, the others are released via `symbol_put()` and GPU features degrade gracefully. The module compiles and loads whether or not the NVIDIA driver is present.

**Replacing `ioremap` with `ioremap_wc` produced a 230x speedup for host-to-GPU writes.** Plain `ioremap` creates an uncacheable mapping where every CPU store becomes a single 8-byte PCIe write TLP. The benchmark measured UC memcpy at 44 MB/s host-to-GPU and 6 MB/s GPU-to-host, against `cudaMemcpy` at 12.5 GB/s and 13.1 GB/s. `ioremap_wc` tells the CPU to buffer stores and flush them as 64-byte cache-line-sized TLPs. Host-to-GPU jumped to 10,097 MB/s -- within 20 percent of `cudaMemcpy`. GPU-to-host improved to 107 MB/s, still 120x slower. This asymmetry is fundamental to PCIe: writes are posted (fire-and-forget), reads are non-posted (the CPU waits for each round-trip). Write-combining helps writes dramatically but cannot fix the read latency. Only a hardware DMA engine can pipeline dozens of outstanding reads to achieve full bandwidth -- the architectural reason GPUDirect RDMA exists.

**BAR pages are mapped individually because contiguity in physical BAR1 space is not guaranteed.** `nvidia_p2p_get_pages` returns an array of 64KB page descriptors at arbitrary BAR1 offsets. A single `ioremap_wc(pages[0], total_size)` would map past the first page into unrelated BAR1 regions. The implementation creates a `bar_pages[]` array with one `ioremap_wc` per 64KB page, then walks it for every `memcpy_toio`/`memcpy_fromio`, computing which page a given offset falls in and clamping each copy to the page boundary. For a 1 MB buffer that is 16 boundary crossings -- negligible overhead compared to the PCIe transfer time.

**The scatter-gather table is constructed from BAR physical addresses directly, bypassing the DMA mapping API.** Normal SG tables carry a `struct page` reference so `dma_map_sg` can program the IOMMU. GPU BAR pages have no `struct page` -- they are MMIO windows, not system RAM. `gpu_bar_to_sgtable` sets `sg_dma_address` and `sg_dma_len` directly to the BAR1 physical address and 64KB. This works because BAR addresses are already visible to any PCIe device on the bus. On rxe there is no hardware IOMMU to program; on ConnectX, `nvidia-peermem` handles IOMMU mapping through `nvidia_p2p_dma_map_pages`. The NULL `struct page` means this table cannot be passed to any kernel function that dereferences the page pointer.

**The unpin callback contract requires atomic-flag-only signaling with lazy cleanup to avoid deadlocking against NVIDIA's internal locks.** When `cudaFree` is called on pinned memory, NVIDIA invokes the `free_callback` while holding internal locks. Calling `nvidia_p2p_put_pages` from the callback would reacquire the same locks -- deadlock. Taking any mutex or sleeping lock is forbidden; even `printk` is unsafe. The callback does two things: atomically sets `gpu_revoked = 1` and signals `revoke_done` via `complete_all`. Actual cleanup happens lazily in `dmaplane_gpu_unpin`: if not revoked, `nvidia_p2p_put_pages` releases the pin normally; if revoked, `nvidia_p2p_free_page_table` just kfrees the struct without touching NVIDIA state. This dual-path teardown is the same pattern `nvidia-peermem` uses, and mishandling the callback is the primary source of deadlocks in third-party GPUDirect RDMA drivers.

**GPU Memory Regions share the same `mrs[]` array, ID namespace, lock, and deregister path as host MRs.** `dmaplane_gpu_register_mr` finds a free slot under `mr_lock`, sets `sge_addr` to the GPU buffer's contiguous WC mapping, and uses `pd->local_dma_lkey` instead of allocating a per-MR hardware key. The `mr` field is NULL (no `ib_alloc_mr` since that requires `struct page`), `rkey` is zero. A separate SG table is built -- not shared with `gpu_buf->sgt` -- because the deregister path frees the MR's SG table and sharing would double-free. From deregister's perspective, a GPU MR looks identical to a host MR. This lets the loopback benchmark send from a GPU MR on QP-A and receive into a host MR on QP-B using the same ioctl interface.

**The peer QP enables cross-machine RDMA by adding a third QP that connects to a remote machine instead of looping back locally.** `rdma_engine_init_peer` creates `qp_peer` and `cq_peer`, transitions to INIT, and returns local metadata: QP number, LID, GID, and Ethernet MAC. Userspace exchanges this over TCP. `rdma_engine_connect_peer` completes the state machine: INIT to RTR with the remote QP number, GID, and DMAC, then RTR to RTS. The DMAC is the subtle detail -- without populating `ah_attr.roce.dmac` with the remote MAC, the Ethernet frame goes nowhere. With `rdma_cm` this happens automatically through ARP; with raw kernel verbs you populate it explicitly from the TCP exchange. The peer QP reuses `qp_to_init`, `qp_to_rtr`, and `qp_to_rts` unchanged from loopback.

**The measured performance hierarchy reveals why each tier of hardware access exists: UC at 44 MB/s, WC at 10 GB/s, cudaMemcpy at 12.5 GB/s, GPUDirect RDMA at 12-24 GB/s.** Each tier uses a smarter DMA strategy. UC sends individual 8-byte TLPs. WC coalesces into 64-byte TLPs. `cudaMemcpy` pipelines dozens of outstanding requests through the GPU's DMA engine. GPUDirect RDMA does the same through the NIC's DMA engine, reading GPU BAR pages at full PCIe bandwidth. For reads, the asymmetry persists: WC achieved 107 MB/s versus `cudaMemcpy` at 13.1 GB/s. The cross-machine demo proved the full path at 11.6 MB/s over 1 GbE. On ConnectX with `nvidia-peermem`, the NIC bypasses the CPU entirely, making the network link the bottleneck. For disaggregated inference transferring gigabytes of KV-cache per second, the CPU path is not slow -- it is unusable.
