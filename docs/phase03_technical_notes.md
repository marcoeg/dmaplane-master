Phase 3 adds dma-buf export as a separate subsystem alongside the existing buffer manager. Where Phase 2 gave buffers a lifecycle — create, mmap, destroy — Phase 3 gives them a second identity: a dma-buf file descriptor that other kernel devices can attach to and map through their own IOMMU contexts. The buffer pages stay exactly where they are. What changes is who can see them.

**The export context is the bridge between the dmaplane buffer and the dma-buf framework.** `struct dmaplane_dmabuf_ctx` is allocated per export and stored in `dmabuf->priv`, the opaque pointer that every `dma_buf_ops` callback receives. It holds a back-pointer to the `dmaplane_dev` singleton (`dev`), a pointer to the backing `dmaplane_buffer` (`buf`), and then three fields borrowed from that buffer: the `pages` array, `nr_pages`, and `vaddr`. These are not copies — they point directly into the buffer's own storage. The borrowing is safe because the `dmabuf_exported` flag on the buffer prevents `dmabuf_rdma_destroy_buffer` from freeing the buffer while a dma-buf wraps it (`-EBUSY`). The context also carries an `attach_count` (atomic) that tracks how many devices are currently attached, purely for diagnostics.

**The `dma_buf_ops` vtable has ten callbacks, but only two do heavy lifting.** `attach` and `detach` are bookkeeping: they increment and decrement the atomic `attach_count` on the context and the device-level stats counters (`dmabuf_attachments`, `dmabuf_detachments`). No filtering, no IOMMU group checks — any device can attach. `begin_cpu_access` and `end_cpu_access` are stubs that return 0 immediately. On x86, hardware cache coherency means the CPU and device see consistent data without explicit sync. On ARM or RISC-V these would call `dma_sync_sg_for_cpu` and `dma_sync_sg_for_device`, but the interface is correct even though x86 doesn't need the work. `vmap` returns the existing kernel virtual address from the buffer's page-backed allocation path — no second `vmap` call, because the mapping's lifetime is the buffer's, not the dma-buf's. `vunmap` is intentionally empty for the same reason: freeing the mapping here would leave the buffer's `vaddr` dangling. `mmap` runs a `vm_insert_page` loop identical to the one in `dmaplane_mmap` for page-backed buffers, with the same `VM_DONTCOPY | VM_DONTEXPAND | VM_DONTDUMP` flags. `release` is discussed below. That leaves `map_dma_buf` and `unmap_dma_buf` — the real work.

**The map callback builds a per-device scatter-gather table in three steps.** First, `kzalloc` allocates an `sg_table`, then `sg_alloc_table` sizes its scatterlist array to `nr_pages` entries. Second, a `for_each_sgtable_sg` loop calls `sg_set_page` for each page in the context's borrowed page array — one SG entry per page, each carrying `PAGE_SIZE` bytes at offset 0. Third, `dma_map_sgtable` programs IOMMU entries for the importing device. This is why the SG table is built at map time, not at export time: each importer may sit behind a different IOMMU with a different DMA address space. Building at export would assume a single IOMMU context — wrong for cross-device sharing. After `dma_map_sgtable`, each SG entry's `dma_address` is valid for that specific device's DMA context. On x86 without IOMMU, DMA addresses equal physical addresses, but the indirection is architecturally correct. The returned `sg_table` is owned by the caller (the dma-buf framework caches it per-attachment). `unmap_dma_buf` reverses the operation: `dma_unmap_sgtable` tears down IOMMU entries, `sg_free_table` frees the scatterlist, `kfree` frees the `sg_table` container.

**Ownership has three layers, each with its own reference count.** The first layer is the exporter reference: `dmaplane_dmabuf_export` sets `buf->dmabuf_exported = true` under `buf_lock`, which prevents the buffer from being destroyed. The second is the dma-buf file reference: `dma_buf_export` creates the `struct dma_buf`, and `dma_buf_fd` installs it in the process's file descriptor table, giving userspace a reference. Every `close(fd)` drops one file ref. The third layer is attachment references: each `dma_buf_attach` pins the dma-buf until the matching `dma_buf_detach`. When the last file reference drops (all fds closed, all kernel `dma_buf_put` calls balanced), the framework invokes the `release` callback, which is when the export context is torn down.

**The release callback clears the exported flag under `buf_lock`.** It acquires `dev->buf_lock`, sets `buf->dmabuf_exported = false` and `buf->dmabuf = NULL`, then releases the lock and frees the context with `kfree`. The lock acquisition is legal because `release` runs in process context — it is called from `fput`, which happens when the last `close(2)` completes or when a `dma_buf_put` drops the final kernel reference. There is no deadlock risk with `dmabuf_rdma_destroy_buffer` because destroy checks `dmabuf_exported` and returns `-EBUSY` without blocking on any lock that `release` holds. After `release` clears the flag, the buffer is once again destroyable.

**Coherent buffers cannot be exported.** The export path checks `buf->alloc_type != DMAPLANE_BUF_TYPE_PAGES` and returns `-EINVAL`. The reason is structural: `dma_alloc_coherent` returns a kernel virtual address and a DMA handle, but no `struct page` array. The SG table construction in `map_dma_buf` calls `sg_set_page`, which needs actual `struct page` pointers. Calling `virt_to_page` on `dma_alloc_coherent` memory is unreliable across architectures — especially with IOMMU or SWIOTLB, where the virtual address may not correspond to a normal page. Page-backed buffers, allocated through `alloc_page` with an explicit page array, have exactly the data `sg_set_page` needs.

**The fd lifecycle has a specific sequence.** `dmaplane_dmabuf_export` calls `DEFINE_DMA_BUF_EXPORT_INFO(exp_info)` to declare the export descriptor on the stack. It fills in the ops pointer, the buffer size, the context as `priv`, and critically sets `exp_info.flags = O_RDWR`. The `O_RDWR` is not optional: without it, the dma-buf file is opened read-only, and any `mmap` attempt with `PROT_WRITE` fails with `-EACCES`. `dma_buf_export` creates the `struct dma_buf` and its backing `struct file`. Then `dma_buf_fd(dmabuf, O_CLOEXEC)` allocates a file descriptor, installs the dma-buf's file in the fd table, and returns the number. `O_CLOEXEC` ensures the fd does not leak across `exec`. If `dma_buf_fd` fails (fd table exhausted), the code calls `dma_buf_put` to release the dma-buf — the `release` callback fires, but `dmabuf_exported` is still false at that point, so the flag-clear is a harmless no-op. Only after both `dma_buf_export` and `dma_buf_fd` succeed does the code set `buf->dmabuf_exported = true`. One dma-buf per buffer — a second export attempt returns `-EBUSY`.

**Locking during export is coarse and deliberate.** `buf_lock` is held for the entire `dmaplane_dmabuf_export` call: buffer lookup, type check, exported-flag check, context allocation, `dma_buf_export`, `dma_buf_fd`, and flag-set. Both `dma_buf_export` and `dma_buf_fd` may sleep (GFP_KERNEL allocations, fd table manipulation), but `buf_lock` is a mutex, so sleeping is permitted. The coarse granularity prevents a race where two threads export the same buffer simultaneously — both would pass the `dmabuf_exported` check and create two dma-bufs for the same backing pages. Exports are rare administrative operations, not hot-path, so the serialization cost is irrelevant.

**What the code doesn't do yet is the point of the next phases.** There is no RDMA memory region registration from the page array — Phase 4 will build MRs using `ib_reg_mr` over these same pages, giving the NIC direct DMA access to the buffer content. There is no NUMA-aware allocation — Phase 5 upgrades `alloc_page` to `alloc_pages_node` so buffers land on specific NUMA nodes, reducing cross-socket memory traffic. And there is no GPU BAR page import — Phase 8 will use the dma-buf importer side to map GPU VRAM into the same framework. The export machinery is in place. The consumers come next.
