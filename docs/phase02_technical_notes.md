Phase 2 adds DMA buffer management as a separate subsystem from the channel machinery. Where Phase 1 built the command-processing pipeline (rings, workers, ioctl dispatch), Phase 2 builds the memory that those commands will eventually operate on. Buffers are global resources on `dmaplane_dev`, not per-channel — any fd can create, mmap, and destroy them. The two subsystems share the device context but don't interact directly yet. They converge in later phases when the worker starts posting real DMA transfers instead of incrementing payloads.

**The platform device is the first thing that had to change.** Phase 1's `device_create` produces a `/dev/dmaplane` node, but the underlying `struct device` has no bus registration and no DMA mask. The DMA API — `dma_alloc_coherent`, `dma_map_sg`, `dma_mmap_coherent` — requires a device that the DMA subsystem can program IOMMU entries for. The fix is `platform_device_alloc("dmaplane_dma")` followed by `platform_device_add` and `dma_set_mask_and_coherent` (try 64-bit, fall back to 32-bit). This happens before the char device registration in `dmaplane_init`. All DMA calls use `&dev->pdev->dev` — the platform device, never the char device.

**A buffer is tracked by `struct dmaplane_buffer`.** It has an `id` (the handle returned to userspace — monotonically increasing, never 0, never reused within a module load), an `alloc_type` (`BUF_TYPE_COHERENT` or `BUF_TYPE_PAGES`), and a `size`. The `in_use` flag tells the linear-scan allocator whether the slot is free. Both paths produce a `vaddr` — a kernel virtual address — but through different mechanisms. For coherent buffers, `vaddr` comes directly from `dma_alloc_coherent` and is physically contiguous; the companion `dma_handle` holds the DMA address for the device. For page-backed buffers, `vaddr` comes from `vmap` over an array of scattered `struct page` pointers, and `dma_handle` is zero. The `mmap_count` (atomic) tracks how many userspace mappings exist and gates destruction. Before populating any of these fields, the code `memset(buf, 0, sizeof(*buf))` — clearing stale pointers from whatever previously occupied the slot.

**The coherent path is simple.** `dma_alloc_coherent(&dev->pdev->dev, size, &dma_handle, GFP_KERNEL)` returns physically contiguous, cache-coherent memory. The kernel and device see consistent data without explicit sync. The allocation is memset-zeroed because `dma_alloc_coherent` does not guarantee zeroing, and we're about to mmap this into userspace — without zeroing, stale kernel data leaks. This path is intended for small control structures (CQ entries, doorbells) where coherence matters more than size.

**The page-backed path does more work.** First, `kvcalloc(nr_pages, sizeof(struct page *), GFP_KERNEL)` allocates the pointer array. Then a loop calls `alloc_page(GFP_KERNEL | __GFP_ZERO)` for each page individually. `__GFP_ZERO` is mandatory — without it, the recycled physical pages carry whatever data the previous owner left behind, and that data becomes visible to userspace through mmap. If any page allocation fails, the error path frees all previously allocated pages in reverse (the standard kernel unwind pattern). After all pages are allocated, `vmap(pages, nr_pages, VM_MAP, PAGE_KERNEL)` creates a contiguous kernel virtual mapping over the scattered physical pages. Without vmap, you'd need `kmap` per page — vmap gives you a single pointer that spans the entire buffer. This vaddr is what the RDMA engine will use in Phase 4 as the SGE address for rxe's memcpy-based send path. If vmap fails, the pages are freed and the error propagates.

**The mmap handler completes the zero-copy chain.** Userspace calls `IOCTL_GET_MMAP_INFO` to get the offset (`buf_id << PAGE_SHIFT`) and size, then calls `mmap(NULL, size, PROT_READ|PROT_WRITE, MAP_SHARED, fd, offset)`. The driver's `dmaplane_mmap` extracts the buffer ID from `vma->vm_pgoff`, finds the buffer under `buf_lock`, and sets three VMA flags: `VM_DONTCOPY` (no COW to forked children — a child with a stale mapping of pages being DMA-written would corrupt data), `VM_DONTEXPAND` (no mremap beyond buffer bounds), and `VM_DONTDUMP` (exclude from core dumps — DMA memory may be sensitive, and dumping would race with device writes). For page-backed buffers, a `vm_insert_page` loop maps each physical page into the VMA one at a time — the pages are scattered and can't be covered by a single `remap_pfn_range`. For coherent buffers, `dma_mmap_coherent` handles the mapping, but with a gotcha: `dma_mmap_coherent` interprets `vm_pgoff` as a page offset into the coherent allocation, not as a buffer ID. Since we encoded the buffer ID there, it sees a non-zero offset that exceeds the allocation and returns `-ENXIO`. The fix is `vma->vm_pgoff = 0` before the call.

**The `mmap_count` increment is in the mmap handler itself, not in `vma_open`.** This is the most subtle detail. Linux does not call `vma_open` for the initial mmap — only for fork and mremap. If you only increment in `vma_open`, the first mapping is untracked, and the destroy path won't block on it. The decrement happens in `vma_close`, which does fire for munmap and process exit. With `VM_DONTCOPY` blocking fork and `VM_DONTEXPAND` blocking mremap, the `vma_open` callback is purely defensive.

**The locking is one mutex.** `buf_lock` protects the entire buffer array: slot allocation, lookup, deallocation, and mmap (the buffer must not be destroyed while `vm_insert_page` is in progress). It's a mutex because all callers are in process context (ioctl and mmap handlers) where sleeping is permitted. It is independent of `dev_mutex` — the two protect different resources and are never nested. In Phase 4, the ordering will extend: `rdma_sem` (read) before `buf_lock` when both are needed.

**The ioctl layer has four new commands.** `CREATE_BUFFER` copies params from userspace, calls `dmabuf_rdma_create_buffer`, and copies the result back. If `copy_to_user` fails after the buffer is created, the handler destroys the buffer — without this, the allocation is orphaned because userspace never received the handle. `DESTROY_BUFFER` takes a bare `__u32` (not a struct) because only the handle is needed. It returns `-EBUSY` if `mmap_count > 0`. `GET_MMAP_INFO` looks up the buffer under `buf_lock` and returns `buf_id << PAGE_SHIFT` as the mmap offset. `GET_BUF_STATS` reads atomic counters into the UAPI struct.

**Module exit ordering matters.** Channels are stopped first (kthreads must exit). Then the char device is torn down — `device_destroy`, `class_destroy`, `cdev_del`, `unregister_chrdev_region` — which prevents any new opens or ioctls from racing with cleanup. Then remaining buffers are destroyed, with a `pr_warn` if any have `mmap_count > 0` (leak detector). Buffer destruction must happen while the platform device is still alive, because `dma_free_coherent` needs it. Only after all buffers are gone does `platform_device_unregister` run. Then `kfree`.

**What the buffers can't do yet is the point.** There is no NUMA placement — Phase 3 upgrades `alloc_page` to `alloc_pages_node` with `page_to_nid` verification. There are no scatter-gather tables — Phase 4 builds them from the page array during MR registration. There is no dma-buf export — Phase 5. There is no RDMA — Phase 4 posts send/recv work requests using these pages. And nothing on the device side reads the data yet. But the pages exist, the kernel can access them through vmap, userspace can write to them through mmap, and the lifecycle — create, map, use, unmap, destroy — is safe against concurrent access and process exit. The memory is ready. The transport comes next.
