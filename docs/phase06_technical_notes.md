Phase 6 adds credit-based flow control to the RDMA engine, a sustained streaming benchmark with per-second throughput windowing, and a queue depth sweep that characterizes the concurrency-throughput curve.

**Completion Queue overflow is catastrophic and silent.** When a sender posts work requests faster than completions are drained, the CQ fills. On real hardware (ConnectX, Efa, hfi1), the firmware transitions the CQ to error state and the QP follows — all in-flight operations are lost, the QP is unrecoverable, and the only remedy is full teardown and re-creation. On rxe, completions are silently dropped, but the effect is equivalent: the sender believes operations completed when they did not. The critical insight is that there is no backpressure from CQ to SQ. The send queue accepts work requests as long as it has free slots, regardless of CQ capacity. The SQ and CQ are decoupled by design — the NIC posts completions asynchronously after the wire transfer finishes — so the sender has no synchronous signal that the CQ is approaching its limit. The only defense is to self-throttle based on how many operations remain in-flight.

**The credit system makes CQ overflow impossible by construction.** `flow_control.c` maintains an `atomic_t credits` inside `dmaplane_dev.flow`, initialized to `max_credits` at configuration time. Every `rdma_engine_post_send` is followed by `dmaplane_flow_on_send` (atomic decrement). Every send completion polled from CQ-A triggers `dmaplane_flow_on_completion` (atomic increment). Before posting, the loop calls `dmaplane_flow_can_send`, which returns false when credits are exhausted. The invariant: in-flight operations never exceed `max_credits`. Because `max_credits` is validated to be at most 128 — the CQ depth — and each in-flight operation generates exactly one CQE, the CQ always has room. This is not a heuristic. The counter makes it algebraically impossible to exceed capacity.

**Watermark hysteresis prevents send-stall oscillation.** Without hysteresis, the sender would hit the credit limit, stall for one completion, post one send, hit the limit, stall again — toggling on every operation, with `cond_resched` overhead dominating throughput. The solution is two thresholds: `high_watermark` (typically 75% of `max_credits`) and `low_watermark` (typically 25%). When in-flight count reaches `high_watermark`, `dmaplane_flow_can_send` sets a `paused` flag. The sender stalls, calling `cond_resched` each iteration to yield CPU time and let the rxe softirq process completions. The flag remains set as completions trickle in — the sender resumes only when in-flight drops to `low_watermark`, at which point a burst of sends refills the pipeline. The gap between thresholds defines the hysteresis band. The test suite exercises both extremes: `max_credits=64`, `high=48`, `low=16` sustains 1037 MB/s with zero overflows, while `max_credits=4`, `high=3`, `low=1` generates 72.7 million credit stalls over 5 seconds and 660 thousand watermark events. The tight configuration proves the mechanism holds under extreme pressure — the off-by-one distance between high and low is a near-terminal condition for throughput, yet the CQ never overflows.

**The sustained streaming benchmark measures wall-clock behavior, not iteration count.** Phase 4's streaming benchmark runs a fixed number of iterations, biasing results toward the steady-state hot path. `dmaplane_sustained_stream` runs for `duration_secs` wall-clock seconds, capturing warmup, pipeline fill, steady state, and drain. Per-second windowing tracks stability: a `window_bytes` counter accumulates transfers, and when elapsed time exceeds `WINDOW_NS` (one second), window throughput is computed, compared against running min/max, and the counter resets. After the main loop, a drain phase polls CQ-A for up to `DRAIN_TIMEOUT_MS` (5 seconds) to collect completions for operations still in-flight when the deadline expired — without this, stale completions would poison the next benchmark. The drain counts toward `total_bytes`, so average throughput reflects all transferred data. A 10-second run with 4 KB messages at queue depth 8 sustained 1037 MB/s average, with per-second windows ranging from 1047 to 1087 MB/s — a 3.8% spread confirming steady-state stability and zero CQ overflows.

**The queue depth sweep reveals where concurrency stops helping.** `dmaplane_qdepth_sweep` iterates from `min_qdepth` to `max_qdepth` in steps, running fixed-iteration streaming at each point. Per-completion latency is recorded via `ktime_get` into a `kvcalloc`-allocated array, sorted with `rdma_engine_cmp_u64` to extract P99 at index `(completed * 99) / 100`. Between points, both CQs are flushed with `rdma_engine_flush_cq` to prevent cross-contamination. Saturation detection uses a 95% threshold: the sweep finds the first point (excluding the last) where throughput reaches 95% of the maximum observed. If only the final point hits the threshold, the curve is still climbing and `saturation_qdepth` reports 0. On the development laptop, QD=1 at 20 MB/s climbed to QD=29 at 567 MB/s with no saturation — expected for rxe, where deeper queues amortize softirq scheduling, SKB allocation, and loopback routing overhead. On real hardware, the curve would flatten at QD=4-8 as PCIe bandwidth saturates.

**Shared utilities were refactored to avoid duplication.** Phase 4's `benchmark.c` contained `flush_cq` and `cmp_u64` — both needed by `flow_control.c`. Rather than copying them, both were moved to `rdma_engine.c` as `rdma_engine_flush_cq` and `rdma_engine_cmp_u64`, declared in `rdma_engine.h`, and exported with `EXPORT_SYMBOL_GPL`. `rdma_engine_flush_cq` loops `ib_poll_cq` until it returns zero or negative. `rdma_engine_cmp_u64` provides the standard three-way `__u64` comparison for `sort`. `benchmark.c` now calls the shared versions.

**What the code does not add connects to what production systems do differently.** There is no CQ moderation (`ib_modify_cq` with `cq_count`/`cq_period`) because rxe has no hardware interrupt coalescing — completions are polled explicitly with `IB_POLL_DIRECT`. There is no multi-QP flow control; the loopback pair is a single stream, while NVMe-oF and iSER manage per-connection credit windows across hundreds of QPs with shared pools and fairness arbitration. There are no receiver-initiated credit grants — in production protocols (NVMe-oF capsule commands, iSER target flow control, NCCL ring-reduce credits), the receiver explicitly tells the sender how many operations it can absorb via out-of-band messages. In dmaplane's loopback topology, credits are local because the sender polls CQ-A directly. Despite these simplifications, the pattern is universal: NVMe-oF advertises `MAXCMD` (structurally identical to `max_credits`), iSER tracks a per-connection command window, and NCCL maintains per-channel credit counters to prevent a fast GPU from overwhelming a slow peer. The watermark hysteresis is equally standard — TCP's receive window, NVMe's command slot re-advertisement, and NCCL's batch credit returns all use the same two-threshold band to prevent oscillation.
