Phase 4 turns dmaplane into a kernel-space RDMA client by registering with ib_core, creating a loopback queue pair, and running send/recv benchmarks entirely within the module — no userspace verbs library, no remote peer.

**The module is its own RDMA consumer.** Phases 1 through 3 built a DMA buffer subsystem that allocates pages, vmaps them, and exports them as dma-bufs. Phase 4 takes those same pages and makes them accessible to the RDMA subsystem. The driver calls `ib_device_get_by_name` to open a Soft-RoCE device (rxe), allocates a Protection Domain, creates two Completion Queues and two Queue Pairs, wires them into a loopback connection, registers Memory Regions from the buffer page arrays, and posts send/recv work requests — all from kernel space. The userspace interface is eight new ioctls: setup, teardown, register MR, deregister MR, loopback test, ping-pong benchmark, streaming benchmark, and RDMA stats. The benchmark code lives in a separate `benchmark.c` to keep the RDMA engine focused on resource lifecycle and the benchmark logic focused on measurement patterns.

**The resource hierarchy is strict: PD scopes everything.** `rdma_engine_setup` builds the stack in dependency order: IB device reference, then PD via `ib_alloc_pd`, then CQ-A and CQ-B via `ib_alloc_cq`, then QP-A (bound to CQ-A for both send and recv completions) and QP-B (bound to CQ-B), and finally the loopback connection that drives both QPs through the RESET, INIT, RTR, RTS state machine. The PD is the root — all CQs, QPs, and MRs created under it share access permissions and are isolated from other protection domains. Both QPs use Reliable Connected (RC) transport, the same type NCCL uses for GPU-to-GPU gradient aggregation. RC guarantees in-order, reliable delivery. `IB_SIGNAL_ALL_WR` means every work request generates a completion queue entry, which simplifies polling at the cost of CQ bandwidth — a deliberate trade-off for a learning driver where debuggability matters more than throughput.

**GID selection is not as simple as "use index 0."** Soft-RoCE exposes multiple GID table entries, all typed `IB_GID_TYPE_ROCE_UDP_ENCAP`. GID[0] is typically the EUI-64 MAC-derived link-local address, but on Ubuntu systems with stable-privacy addressing enabled, this address is a ghost — it does not exist on the physical interface. QP transitions will succeed, but traffic fails silently. `scan_gid_table` walks up to 16 entries, filters for non-zero link-local RoCEv2 GIDs, and picks the last one found, which empirically corresponds to the real interface address. If nothing suitable appears, it falls back to index 0 with a warning. The selected GID and its index are stored in `dmaplane_rdma_ctx` and reused for every QP state transition and address handle.

**The loopback pair exists because RDMA resources require a connected QP to function.** QP-A is the sender, QP-B is the receiver. They are connected to each other through `connect_loopback_qps`, which drives each QP through the three-step state machine: `qp_to_init` sets the port number, partition key, and access flags; `qp_to_rtr` (Ready-to-Receive) sets the path MTU, destination QP number (each points to the other), and populates the address handle with the GID, GID index, sgid_attr, and destination MAC; `qp_to_rts` (Ready-to-Send) sets the timeout, retry counts, and packet sequence number. The destination MAC for loopback is the interface's own MAC address, extracted inside an RCU read section via `rdma_read_gid_attr_ndev_rcu`. The ndev reference is only valid inside the RCU section, so the MAC is copied into a stack buffer before releasing. Without the correct DMAC in the address handle, RoCEv2 packets are malformed at the Ethernet layer and silently dropped.

**IB_POLL_DIRECT is a deliberate architectural choice.** CQs are created with `ib_alloc_cq(IB_POLL_DIRECT)`, which means completions are polled explicitly by calling `ib_poll_cq` in a busy-wait loop rather than being delivered through softirqs or workqueues. `rdma_engine_poll_cq` implements this: it loops calling `ib_poll_cq`, yielding with `cond_resched` and sleeping 50-200 microseconds between polls, until either a completion arrives or a wall-clock deadline expires. The trade-off is CPU cost during polling, but the payoff is deterministic, race-free processing. There are no background callbacks that could fire during teardown, no asynchronous state changes, and no races with module unload. The managed CQ layer still requires every `wr_cqe.done` to point to a valid function, so `poll_cq_done` exists as a safety net — it stashes the work completion and signals a `struct completion` — but under normal operation it is never invoked.

**MR registration has two paths, selected by access flags.** When `access_flags` contains only `IB_ACCESS_LOCAL_WRITE`, the driver uses `pd->local_dma_lkey` — a per-PD shortcut that authorizes the local NIC to DMA local pages without allocating a real MR. The rkey is zero (no remote access), and `sge_addr` is set to the kernel virtual address because rxe (Soft-RoCE) interprets `sge.addr` as a kernel VA via the local_dma_lkey path. When access flags include remote permissions (`IB_ACCESS_REMOTE_WRITE`, `IB_ACCESS_REMOTE_READ`), the driver must allocate a fast-reg MR via `ib_alloc_mr` with type `IB_MR_TYPE_MEM_REG`, map the scatter-gather list onto it with `ib_map_mr_sg`, and post an `IB_WR_REG_MR` work request on QP-A to activate the MR in the NIC's memory translation tables. The resulting `frmr->lkey`, `frmr->rkey`, and `frmr->iova` replace the local_dma_lkey values. Both paths start the same way: `register_mr_build_sgt` allocates an `sg_table` with one entry per page (the pages are order-0 and not physically contiguous), populates each scatterlist entry with `sg_set_page`, and calls `ib_dma_map_sg` to establish IOMMU mappings for the full buffer. The buffer's page array is snapshotted into `pages_copy` under `buf_lock` — a kvmalloc'd copy — so that releasing `buf_lock` before the RDMA operations does not risk use-after-free if another thread destroys the buffer concurrently.

**rdma_sem is the outer gate; buf_lock and mr_lock are the inner resource locks.** `rdma_sem` is a read-write semaphore on the device struct. Write lock (`down_write`) is taken by RDMA setup and teardown — exclusive access, no other RDMA operations can proceed. Read lock (`down_read`) is taken by MR registration, MR deregistration, and all three benchmarks — multiple readers run concurrently, guaranteeing the RDMA context stays alive while they execute. The ordering is: `rdma_sem` (read) before `buf_lock` before `mr_lock`. No code path holds `rdma_sem` write while holding `buf_lock` or `mr_lock`. Inside the benchmarks, the MR entry is snapshotted by value under `mr_lock` — the struct is copied, the lock is released, and all subsequent I/O uses the local copy. This decouples benchmark execution from concurrent MR deregistration: even if another thread deregisters the MR while the benchmark is mid-flight, the benchmark's cached lkey and sge_addr remain valid for the duration of that run.

**Teardown is disciplined and order-sensitive.** `rdma_engine_teardown` first transitions both QPs to `IB_QPS_ERR` via `ib_modify_qp`. This is not optional — the RDMA subsystem uses the ERR transition to flush all outstanding work requests. Without it, destroying a QP that has pending WRs leaves the provider holding references to the module's CQE and SGE memory, which leads to use-after-free when the module unloads. After the ERR transition, resources are destroyed in strict reverse order of creation: QP-B, QP-A, CQ-B, CQ-A, PD, IB device reference. Because CQs use `IB_POLL_DIRECT`, there are no background workqueues or softirq handlers to drain — teardown is race-free without explicit CQ draining. The teardown ioctl goes further: before calling `rdma_engine_teardown`, it loops through all MR slots and deregisters every in-use MR, because their SG table DMA mappings hold references to the `ib_dev` that teardown will release. The module exit path does the same, with an additional guard: it only deregisters MRs if `rdma.initialized` is true, preventing a use-after-free on the `ib_mr` if userspace already called the teardown ioctl.

**What the code does not do yet is the point of the later phases.** There are no peer QPs — the loopback pair is self-contained on a single machine. Phase 8 will add cross-machine RDMA by exchanging QP metadata (QPN, GID, LID) between two hosts and connecting their QPs. There is no RDMA WRITE with immediate data — Phase 9 will add `IB_WR_RDMA_WRITE_WITH_IMM` for zero-copy, one-sided transfers where the sender writes directly into the receiver's registered memory and the immediate value signals completion without a separate message. The fast-reg MR path exists now precisely to support these future phases: when QP-A on machine A needs to write into QP-B's memory on machine B, it needs a real rkey, not local_dma_lkey. The infrastructure is in place; the wire protocol is not.
