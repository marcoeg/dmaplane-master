Phase 5 adds NUMA awareness to dmaplane: the ability to steer buffer allocations to a specific NUMA node, query the machine's topology from kernel space, and measure the bandwidth cost of cross-node memory access with an in-kernel benchmark.

**The core addition is NUMA-targeted page allocation.** When userspace passes a node ID in `dmaplane_buf_params.numa_node`, the page-backed allocation path in `dmabuf_rdma.c` calls `alloc_pages_node(nid, GFP_KERNEL | __GFP_ZERO, 0)` instead of the unqualified `alloc_pages`. This function does not guarantee placement. It selects the target node's zonelist as the starting point for the buddy allocator walk, but if the node's free pools are exhausted, the allocator falls back silently to progressively more distant nodes, ordered by the ACPI SLIT distance table. There is no strict mode. The kernel never returns an error just because a page ended up on the wrong socket. The only way to detect misplacement is after the fact, by calling `page_to_nid()` on each returned page and comparing the result against the requested node. The allocation loop does exactly this: it counts local and remote placements page by page, and if any page lands elsewhere, it logs a warning with the mismatch count. The per-buffer `actual_numa_node` field reports the majority-vote result back to userspace. When `DMAPLANE_NUMA_ANY` (-1) is passed, the allocator receives `NUMA_NO_NODE`, which is equivalent to plain `alloc_pages` — it uses the current CPU's local node without any cross-node attempt.

**Coherent allocations cannot be steered.** `dma_alloc_coherent` takes a `struct device *`, not a NUMA node. The DMA subsystem chooses the allocation node based on the device's own NUMA affinity, which for a platform device is typically node 0. There is no `dma_alloc_coherent_node` API in the kernel. So when userspace requests a coherent buffer on node 1, the code ignores the request and allocates wherever the DMA layer decides. It still calls `page_to_nid(virt_to_page(buf->vaddr))` after the fact to report the actual node, but it cannot control the outcome. This asymmetry is deliberate: coherent memory is for small control structures where locality matters less, while page-backed buffers are for the large gradient and weight staging areas where NUMA placement directly affects throughput.

**The NUMA statistics track placement at buffer granularity, not page granularity.** Each `CREATE_BUFFER` increments exactly one of three device-level atomic counters. `numa_local_allocs` means every page in the buffer landed on the requested node. `numa_remote_allocs` means at least one page was misplaced — the allocator fell back. `numa_anon_allocs` means the caller passed `DMAPLANE_NUMA_ANY` and did not care about placement. These counters are exposed to userspace through `IOCTL_GET_BUF_STATS` in the existing `dmaplane_buf_stats` struct, which gained three new `__u64` fields. The per-buffer granularity is a simplification. A production system would want per-page tracking with a bitmap indicating which pages were misplaced, but for a learning project the buffer-level signal is sufficient to detect when the system is under enough memory pressure to cause fallback.

**The topology query exposes the kernel's NUMA view through a single ioctl.** `IOCTL_QUERY_NUMA_TOPO` returns a `dmaplane_numa_topo` struct containing the number of online nodes, per-node CPU counts, per-node total and free memory in kilobytes, and the full NxN distance matrix. Internally, `dmaplane_query_numa_topo` iterates `for_each_online_node` and `for_each_online_cpu` to build the compacted node list (kernel node IDs can be sparse — node 0 and node 2 online with node 1 offline — so a `node_to_idx` translation array maps sparse IDs to contiguous array indices). Per-node memory comes from `node_present_pages()` for the total and a walk of all zones in `NODE_DATA(node)->node_zones` summing `zone_page_state(z, NR_FREE_PAGES)` for the free count. The `managed_zone()` check skips zones that the kernel has not activated. The distance matrix is populated by calling `node_distance(a, b)` for every node pair, which reads the firmware-provided SLIT table: 10 means same-node, 20-21 means one QPI/UPI hop, higher means further. On a single-socket machine the matrix is 1x1 with distance 10. The `node_to_idx` array is heap-allocated via `kvcalloc` because `MAX_NUMNODES` can be as large as 1024 on some kernel configs, too large for the stack.

**The bandwidth benchmark measures the real cost of cross-node access.** `IOCTL_NUMA_BENCH` accepts a buffer size and iteration count, then for every (cpu_node, buf_node) pair among online nodes, it allocates a destination buffer on `buf_node` and a source buffer on `cpu_node`, spawns a kthread, pins it to all CPUs on `cpu_node` via `set_cpus_allowed_ptr`, and runs `iterations` memcpy operations from source to destination. The result is an NxN matrix of bandwidth in MB/s and latency in nanoseconds per iteration. The design makes several deliberate choices. Cells execute sequentially, not in parallel, because concurrent memcpy from multiple node pairs would contend on the memory bus and under-report bandwidth. Kthreads are used instead of workqueues because workqueues do not guarantee which CPU runs the work item — precise affinity control is the entire point. The cpumask includes all CPUs on the source node rather than pinning to a single core, which avoids single-core thermal effects and better models real workloads. Each cell allocates fresh buffers rather than reusing shared ones, to avoid cache warming artifacts from the previous measurement. Before timing begins, both buffers are touched with `memset` to fault in pages and exclude page fault overhead from the measurement. After each `memcpy`, `barrier()` prevents the compiler from optimizing away the write (it might notice the destination is never read) and ensures each iteration's stores complete before the next begins — modeling the real pattern where a NIC would DMA-read the buffer between CPU writes. Bandwidth is computed as `(size * iterations * 1000) / elapsed_ns`, yielding MB/s. On a dual-socket c5.metal (2x Xeon Platinum 8124M, 96 CPUs, 36 MB L3 per socket), the benchmark with 1 MB buffers showed no cross-node penalty — 8015 MB/s local vs 8983 MB/s remote for node 0, with remote actually appearing faster. This is because both 1 MB buffers fit in the 36 MB L3 cache after the `memset` pre-touch, so the benchmark measures cache bandwidth, not DRAM bandwidth. Increasing to 64 MB buffers (exceeding L3) revealed the real penalty: 6778 MB/s local vs 5577 MB/s remote (18% penalty) for node 0, and 6095 MB/s local vs 5013 MB/s remote (18% penalty) for node 1. The implication for production benchmarking is that any NUMA microbenchmark reporting zero cross-node penalty should be suspected of fitting in cache. Gradient buffers and KV-cache blocks are hundreds of megabytes — they will always hit DRAM.

**The ABI changed.** `dmaplane_buf_params` grew two fields: `numa_node` (input, `__s32`) and `actual_numa_node` (output, `__s32`). `dmaplane_buf_stats` grew three fields for the NUMA counters. Two new ioctl numbers were added in the `0x50` range: `IOCTL_QUERY_NUMA_TOPO` (0x50) and `IOCTL_NUMA_BENCH` (0x51). Every existing caller of `CREATE_BUFFER` — the Phase 2 tests, the Phase 3 dma-buf tests, the Phase 4 RDMA tests — needed updating to initialize `numa_node` to `DMAPLANE_NUMA_ANY` in the params struct. Without this, the field would contain stack garbage, and the kernel-side validation (`node_online(target_node)`) would reject the request with `-EINVAL`. This is the classic UAPI growth problem: adding a field to a shared struct breaks all existing callers unless they zero-initialize or explicitly set the new field. Notably, `{0}` initializes the `__s32 numa_node` field to 0 (node 0), not -1 (`DMAPLANE_NUMA_ANY`). On a single-socket development machine this works by accident — node 0 is the only node. On the dual-socket c5.metal used for validation, `numa_node = 0` would silently force all allocations to socket 0 even when the allocating thread runs on socket 1.

**What the code does not do yet.** There is no IRQ affinity steering — the module cannot pin a NIC's interrupt to the same node as the buffer, because there is no real NIC (rxe is software). On real hardware, mismatched IRQ-to-buffer NUMA placement is as costly as mismatched CPU-to-buffer placement, and production systems like NCCL's topology detection exist specifically to align GPU, NIC, and CPU on the same socket. There is also no automatic placement policy. Userspace must explicitly choose a node. A production system would infer the optimal node from the target device's PCI topology — `dev_to_node(&pci_dev->dev)` — and place buffers there without userspace intervention. Both of these are future-phase concerns. For now, Phase 5 provides the measurement infrastructure: if you can query the topology and benchmark the penalty, you have the data to make informed placement decisions, even if those decisions are still manual. Validation on a dual-socket c5.metal (2 NUMA nodes, distance 21, 48 CPUs per socket) confirmed all 8 tests pass, the distance matrix matches `numactl --hardware`, NUMA-pinned allocations land correctly (`actual_numa_node == 0` when requesting node 0), and the 18% bandwidth penalty at 64 MB buffer sizes is consistent and reproducible across load/unload cycles.
