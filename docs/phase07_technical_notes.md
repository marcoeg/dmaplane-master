Phase 7 adds six kernel tracepoints, five debugfs files, and a log2 latency histogram with P50/P99/P999 percentile computation to the driver, giving the complete observability stack needed to diagnose performance at every layer of the data path.

**The `TRACE_EVENT` build pattern requires exactly one compilation unit to own the tracepoint definitions.** `dmaplane_trace.c` defines `CREATE_TRACE_POINTS` before including `dmaplane_trace.h`, which causes the compiler to emit the tracepoint registration structures and the `trace_dmaplane_*()` call stubs. Every other `.c` file includes the same header without `CREATE_TRACE_POINTS`. Defining it in two files causes duplicate symbol errors at link time. Two Makefile requirements make this work for out-of-tree modules: `TRACE_INCLUDE_PATH` must be `.` (dot) so the trace subsystem's `define_trace.h` can re-include the header during its multi-pass expansion, and `ccflags-y += -I$(src)` ensures the Kbuild system resolves that dot-relative path. Without `-I$(src)`, the build succeeds in-tree but fails out-of-tree. One further constraint is version-specific: kernel 6.5 requires the two-argument form `__assign_str(field, src)`, while newer kernels accept a single argument. The six tracepoints cover the core data paths: `dmaplane_ring_submit` and `dmaplane_ring_complete` on the ioctl path, `dmaplane_rdma_post` and `dmaplane_rdma_completion` on the RDMA engine, `dmaplane_buf_alloc` on buffer creation, and `dmaplane_flow_stall` on credit exhaustion during sustained streaming.

**Tracepoint overhead is negligible when disabled and acceptable when enabled.** Each tracepoint compiles to a static branch — a `nop` instruction patched to a `jmp` when tracing is activated via ftrace. The disabled cost is under 1 nanosecond. When enabled, each event writes to the per-CPU trace ring buffer at roughly 100 to 200 nanoseconds. This does not distort rxe measurements because a single RDMA loopback operation takes approximately 200 microseconds — the tracepoint adds 0.05 to 0.1 percent to operation time. The six tracepoints are placed at subsystem boundaries (ioctl entry, RDMA post, CQ poll, flow stall), not inside hot loops, so per-operation cost is bounded to one or two firings per transfer.

**The debugfs lifecycle is deliberately fire-and-forget.** `dmaplane_debugfs_init` runs from module init after all subsystems are initialized, creating `/sys/kernel/debug/dmaplane/` with five read-only files. `dmaplane_debugfs_exit` runs from module exit before device teardown — this ordering is critical because the seq_file show functions dereference `dmaplane_dev` fields, and tearing down the device first would leave dangling pointers. `debugfs_remove_recursive` atomically removes the entire directory tree and handles a NULL root gracefully. debugfs failure is never fatal: if `CONFIG_DEBUG_FS` is disabled or debugfs is not mounted, `debugfs_create_dir` returns `ERR_PTR(-ENODEV)`, the init function sets root to NULL, and the module operates normally. The five files are: `stats` (all device-level atomic counters), `buffers` (active DMA buffer listing with type, size, NUMA placement, mmap and dma-buf state), `rdma` (RDMA context, QP numbers, active MR list), `flow` (flow control configuration and credit/pause state), and `histogram` (latency distribution with percentiles). All use `seq_file` with `DEFINE_SHOW_ATTRIBUTE`. The `rdma` file acquires `rdma_sem` (read) and `mr_lock`; the `buffers` file acquires `buf_lock`.

**The histogram uses log2 buckets for constant-space latency distribution.** Sixteen buckets cover powers of two in microseconds: bucket 0 is [0, 1) microseconds, bucket i is [2^i, 2^(i+1)) microseconds, bucket 15 is the overflow. Bucket assignment converts nanoseconds to microseconds and applies `ilog2`; sub-microsecond samples route to bucket 0 because `ilog2(0)` is undefined. All fields are `atomic64_t` — bucket counters, total count, cumulative sum, min, and max. No lock is needed. The min/max fields use a CAS loop: read current, compare, atomically swap if the new sample is more extreme. Under contention one thread may lose a race, but the loop converges on the next sample. Percentile computation walks buckets 0 through 15, accumulating counts, and reports the bucket upper bound where cumulative fraction crosses P50/P99/P999. This is coarse — the bucket upper bound is the estimate, not an exact median — but it costs zero allocations and runs in constant time. In the sustained streaming benchmark (5 seconds, queue depth 8, 4 KB messages), the histogram showed 80.8 percent of operations completing in the [16, 32) microsecond bucket — the actual RDMA pipe time from `ib_post_send` to CQ completion, not the scheduling-dominated ~200 microsecond round trip.

**The timestamp ring pairs sends with completions for per-operation latency.** Sustained streaming posts sends asynchronously and polls completions later, so no single call site has both start and end timestamps. A 128-entry `ktime_t` ring, allocated via `kvcalloc` at the start of `dmaplane_sustained_stream`, solves this. On each send, `ktime_get()` is stored at `ts_head & (TS_RING_SIZE - 1)` and head advances. On each CQ-A completion, elapsed time is computed from `send_ts[ts_tail & (TS_RING_SIZE - 1)]` and tail advances. The pairing works because RC QP guarantees in-order completion. The ring size of 128 is a power of two for cheap bitmask modulo and exceeds the maximum queue depth, so it never overflows. The ring was originally stack-allocated but 128 entries at 8 bytes each, combined with the function's other locals, pushed the frame past the kernel's `-Wframe-larger-than=` threshold. Moving to heap allocation via `kvcalloc` eliminated the warning.

**What the code does not add defines the boundary between learning and production.** There is no BPF-based tracing — the production approach (bcc, bpftrace) requires a userspace loader that would obscure the tracepoint mechanics themselves. There is no interrupt coalescing tuning because rxe uses `IB_POLL_DIRECT` with explicit polling; on ConnectX, CQ moderation is a critical knob that trades interrupt rate for tail latency. There is no automated anomaly detection. There are no per-QP histograms — the single device-global `rdma_hist` aggregates all benchmark paths, sufficient for one loopback pair but requiring partitioning in a multi-connection system.

**The complete instrumentation stack mirrors production RDMA observability with four complementary layers.** Tracepoints are the event layer — zero-cost when disabled, per-event detail when enabled, queryable via ftrace or perf. debugfs is the live state layer — always available, gives a current snapshot without setup. The histogram is the distribution layer — answers "what does the latency curve look like" without storing individual samples. `perf stat` is the hardware counter layer — connects kernel behavior to microarchitecture (TLB misses with hugepages from Phase 2, cache misses under contention from Phase 1). This is the same layering used in production: `mlx5_core` exposes tracepoints for WQE posting and CQE processing, debugfs files for QP state and firmware diagnostics, RDMA counters via `rdma_counter` for per-port statistics, and `perf` integration for hardware monitoring. The difference is scale and the absence of firmware telemetry, but the architectural pattern — events, state, distributions, hardware counters — is identical.
